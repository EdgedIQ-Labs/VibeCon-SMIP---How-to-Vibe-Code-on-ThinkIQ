Welcome to the playground.

This is the workbench for iterating on SMIP-side scripts without round-tripping through the SMIP IDE on every edit. Two trees on the left, one render frame on the right.

Display scriptsnode-bound twins

Pick a node in the instance tree. If a folder under DISPLAY_SCRIPTS/<folder>/ binds to that node's type, the right pane mounts the twin and passes the node id as std_inputs.node_id — the same shape SMIP gives the script in production. Edit local_twin.html, refresh, see real data.

Browser scriptsstandalone twins

Pick from the Browser Scripts list. No node binding; the twin renders whatever it wants (a library list, a dashboard, anything page-level). Edit BROWSER_SCRIPTS/<folder>/local_twin.html and refresh.

Adding a twin

  • Decide whether it's node-bound (display) or page-level (browser).
  • Create the folder under the matching top-level directory.
  • Drop in __init__.py, component.py, and local_twin.html.
  • For display scripts, add an entry to DISPLAY_SCRIPTS_BY_TYPE in PLAYGROUND/playground.py.
  • For browser scripts, add an entry to BROWSER_SCRIPTS_CATALOG in the same file.
  • Restart the playground.

Exporting a type

Select any node in the tree — the details panel has an "Export type to SMIP Exports/" button that pulls the type's JSON via GraphQL and writes it to ___SMIP_SAAS_SIDE___/SMIP Exports/<type_fqn>.json. That folder is the canonical paste-target for SMIP type imports — committing it tracks your modeling work in source control.
